# SDD: Web Demonstration Dashboard

> Status: Draft

## 1. Overview

A GitHub Pages public dashboard that exposes the FUSE AI agent ecosystem and token usage to external audiences (recruiters, collaborators). The dashboard is a single static HTML file (`demonstration-orchestration.html`) generated automatically by a CI workflow when a PR targets `main`. On merge, the file is published via GitHub Pages. The goal is maximum recruiter visibility at minimum token cost — no server, no runtime, no recurring processing.

---

## 2. User Stories

- As a **recruiter**, I want to see the project's AI agents and their responsibilities in a public URL, so that I can evaluate the engineering quality without cloning the repository.
- As a **developer**, I want the dashboard to update automatically on every PR to `main`, so that it always reflects the current agent state before merging.
- As a **developer**, I want to see a chart of daily token usage on the dashboard, so that I can monitor the cost efficiency of local vs remote routing.
- As a **team member**, I want the GitHub Pages link embedded in the README, so that visitors land on the live dashboard with one click.

---

## 3. Dashboard Structure

The `demonstration-orchestration.html` file is a single-page static document with three sections:

### Section 1 — Agent Directory

A card grid listing every agent discovered from `.ai/agents/*.md`:
- Agent name (derived from filename)
- Role (extracted from `## 🎯 Role` heading in the `.md`)
- LLM model (`Local` / `Claude`) extracted from `## 🤖 LLM` heading
- Badge: `Always Local`, `Always Remote`, or `Conditional`

### Section 2 — Token Usage Chart (Phase 1)

A bar or line chart (Chart.js, embedded via CDN) showing daily total token consumption, sourced from `.ai/router/token-usage.csv`. Aggregated to `date → total_tokens`.

### Section 3 — Router Summary

A static description of the routing strategy (local vs remote) from `.ai/router/router.md`, rendered as readable prose.

### Navigation

No routing — single scrollable page. Three anchor links in the header: `#agents`, `#tokens`, `#router`.

## 4. Domain Model

```typescript
interface Agent {
  name: string;           // e.g. "react-native-engineer"
  filename: string;       // e.g. "react-native-engineer.md"
  role: string;           // text under "## 🎯 Role"
  llmRouting: 'local' | 'remote' | 'conditional';
  model: string;          // e.g. "claude-sonnet-4-6" or "qwen2.5-coder:14b"
}

interface DailyTokenUsage {
  date: string;           // "YYYY-MM-DD"
  totalTokens: number;
  claudeTokens: number;
  ollamaTokens: number;
}

interface DashboardData {
  generatedAt: string;    // ISO timestamp
  repoUrl: string;
  agents: Agent[];
  tokenUsage: DailyTokenUsage[];
}
```

## 5. API Contract

No external API. All data is file-based, generated at CI time.

### Data Sources

| Source | Format | Consumer |
|--------|--------|----------|
| `.ai/agents/*.md` | Markdown | Agent discovery script |
| `.ai/router/token-usage.csv` | CSV (8 columns) | Token aggregation script |
| `.ai/router/router.md` | Markdown | Router summary section |

### Generated Artifact

**`/docs/demonstration-orchestration.html`**
- Generated by: `.ai/scripts/generate-dashboard.sh`
- Triggered by: GitHub Actions on PR → main and on push → main
- Content: Self-contained HTML with embedded JSON data + Chart.js CDN

The script embeds all data as a JSON block inside a `<script id="dashboard-data">` tag so the HTML needs no external API calls at runtime.

---

## 6. DTO → Model Mapping

### CSV → DailyTokenUsage

```
CSV column[0] date_str (YYYY-MM-DD HH:MM:SS)  →  date: date_str[:10]
CSV column[2] provider                          →  group by: claudeTokens / ollamaTokens
CSV column[7] total_tokens                      →  sum → totalTokens
```

Aggregation is done by date + provider in Python (same pattern as `generate-token-md.sh`).

### Agent Markdown → Agent

```
filename (without .md)           →  name
"## 🎯 Role" next paragraph      →  role  (first non-empty line after heading)
"## 🤖 LLM" content             →  model + llmRouting
  contains "always remote"       →  llmRouting: 'remote'
  contains "always local"        →  llmRouting: 'local'
  else                           →  llmRouting: 'conditional'
```

Parsing is done in Python via regex, not a full Markdown parser.

No SQLite. This feature is entirely file-based and CI-driven.

### Files Written

| `/docs/demonstration-orchestration.html` | `generate-dashboard.sh` | PR workflow + merge workflow |

### Offline Strategy

Not applicable. The dashboard is a static file — once generated and pushed to `docs/`, it is served by GitHub Pages without any runtime dependency.

### Script Location

```
.ai/scripts/generate-dashboard.sh   ← main generation script
.github/workflows/dashboard-pr.yml  ← triggers on PR → main
.github/workflows/dashboard-merge.yml ← triggers on push to main, enables Pages
```

---

## 8. State Strategy

No reactive state. The dashboard is static HTML with embedded data.

At generation time, `generate-dashboard.sh` injects all data as a JSON literal inside the HTML:

```html
<script id="dashboard-data" type="application/json">
  { "agents": [...], "tokenUsage": [...], "generatedAt": "..." }
</script>
```

The Chart.js initialization script reads this JSON at page load — no fetch, no state management.

## 9. Script Design

### `generate-dashboard.sh`

```
Input:
  - REPO_ROOT (env or derived from git)
  - Optional: GITHUB_PAGES_URL (defaults to https://<owner>.github.io/FUSE)

Steps:
  1. Scan .ai/agents/*.md → parse Agent[]
  2. Read .ai/router/token-usage.csv → aggregate to DailyTokenUsage[]
  3. Serialize both to JSON
  4. Render HTML template with injected JSON + Chart.js CDN link
  5. Write to /docs/demonstration-orchestration.html

Output:
  /docs/demonstration-orchestration.html (self-contained)
```

### `dashboard-pr.yml` (GitHub Actions)

```yaml
on:
  pull_request:
    branches: [main]

jobs:
  generate-dashboard:
    steps:
      - checkout
      - run: bash .ai/scripts/generate-dashboard.sh
      - uses: actions/upload-artifact
          with: { name: dashboard, path: docs/demonstration-orchestration.html }
```

### `dashboard-merge.yml` (GitHub Actions)

```yaml
on:
  push:
    branches: [main]

jobs:
  publish-dashboard:
    permissions:
      contents: write
      pages: write
      id-token: write
    steps:
      - checkout
      - run: bash .ai/scripts/generate-dashboard.sh
      - run: |
          git add docs/demonstration-orchestration.html
          git commit -m "chore: update dashboard [skip ci]" || true
          git push
      - uses: actions/configure-pages
      - uses: actions/deploy-pages
          with: { path: docs/ }
```

## 10. Error Handling

| Scenario | Handling |
|----------|----------|
| `token-usage.csv` is empty or missing | Render chart with empty dataset and message "No token data yet" |
| Agent `.md` has no `## 🎯 Role` section | Use filename as role fallback |
| GitHub Pages not enabled | `dashboard-merge.yml` uses `actions/configure-pages` which enables it automatically |
| CI workflow fails | PR is blocked; developer sees Actions failure log; no dashboard artifact produced |
| No agents found in `.ai/agents/` | Render empty agent grid with "No agents found" message |

---

## 11. Edge Cases

| Edge Case | Resolution |
|-----------|-----------|
| Agent `.md` filename has a space (e.g. `" frontend-architect.md"`) | Script trims whitespace from filename before using as `name` |
| `token-usage.csv` has partial rows (< 8 columns) | Skip malformed rows silently (same as existing `generate-token-md.sh`) |
| Push to `main` without changes to `.ai/` or CSV | Workflow still runs; `git commit` exits with 0 if nothing changed (`|| true`) |
| First run — `docs/` folder does not exist | Script creates `docs/` with `mkdir -p docs/` before writing |
| GitHub Pages URL not yet known | Use placeholder `https://eugenioduarte.github.io/FUSE/` (hardcoded from repo name) |
| README already has the Pages link | Script checks with `grep` before appending to avoid duplicates |

## 12. Out of Scope

- **Phase 2 token analytics** (per-model breakdown, monthly charts) — implement only when real usage accumulates
- **GitHub Wiki sync** — deferred; requires wiki repository setup
- **Real-time dashboard** — static only; no WebSockets or polling
- **Authentication** — public dashboard, no login required
- **Mobile-responsive design polish** — basic responsive layout only; `ui-polish` agent is out of scope here
- **Ollama token tracking in chart** — Phase 1 shows total only; provider split is Phase 2
- **Automatic PR comment with dashboard preview link** — future enhancement

---

## 13. Implementation Order

1. **Create `docs/` folder** and add `.gitkeep` so GitHub Pages source directory exists
2. **Write `generate-dashboard.sh`** — agent scanner + CSV aggregator + HTML template renderer
3. **Test locally** — run script, open `docs/demonstration-orchestration.html` in browser
4. **Write `dashboard-pr.yml`** — trigger on PR → main, upload artifact
5. **Write `dashboard-merge.yml`** — trigger on push → main, commit + deploy to Pages
6. **Enable GitHub Pages** via repo settings (source: `docs/` folder, branch: `main`) — or let the workflow do it automatically
7. **Update README** — add `## 🤖 AI Agents Dashboard` section with Pages URL
8. **Validate end-to-end** — open a PR to `main`, confirm artifact is generated; merge, confirm Pages is live

---

## 14. Quality Gates

- `generate-dashboard.sh` exits 0 on empty CSV and empty agents directory
- Generated HTML passes W3C validation (no broken tags, valid `<script>` injection)
- Chart.js loads from CDN — no local bundle needed; no build step required
- GitHub Actions workflow completes in < 60 seconds (static generation only)
- README contains exactly one GitHub Pages link (idempotent script)
- `docs/demonstration-orchestration.html` is committed and visible at the GitHub Pages URL after merge

## 🎯 Request Classification Rules

### Classification Logic

```typescript
function classifyRequest(userInput: string): AgentType {
  // Architecture & coupling keywords
  if (
    matches(userInput, [
      'sdd',
      'architecture',
      'design',
      'structure',
      'refactor strategy',
      'coupling',
      'dependency graph',
      'analyze-coupling',
    ])
  ) {
    return 'architect'
  }

  // Implementation keywords
  if (
    matches(userInput, [
      'implement',
      'create',
      'build',
      'add feature',
      'component',
      'hook',
      'logic',
    ])
  ) {
    return 'engineer'
  }

  // Testing keywords
  if (matches(userInput, ['test', 'e2e', 'coverage', 'unit test', 'flow'])) {
    return 'test-writer' // handles both unit and E2E modes
  }

  // Review & fix keywords
  if (matches(userInput, ['review', 'fix-pr', 'pr comment'])) {
    return 'reviewer'
  }

  // Quality keywords (Sonar + Performance)
  if (
    matches(userInput, [
      'sonar',
      'fix-sonar',
      'quality gate',
      'performance',
      'audit',
      'scan',
      'fps',
      'memory',
      'tti',
    ])
  ) {
    return 'quality'
  }

  // Design & docs keywords
  if (
    matches(userInput, [
      'ui-polish',
      'update readme',
      'business-to-sdd',
      'summary',
      'design system',
    ])
  ) {
    return 'design-docs'
  }

  // PR lifecycle
  if (matches(userInput, ['pr-lifecycle', 'pull request lifecycle'])) {
    return 'pr-lifecycle'
  }

  // Git keywords
  if (matches(userInput, ['commit', 'push', 'branch', 'pr', 'pull request'])) {
    return 'SYSTEM_DIRECT' // Handle directly
  }

  // No match found
  return 'CREATE_NEW_AGENT'
}
```

## 🚨 Critical Orchestration Rules

### Rule 1: Never Bypass Agent Delegation

❌ **Wrong:**

```
User: "create a dashboard component"
System: [directly implements without loading react-native-engineer]
```

✅ **Correct:**

```
User: "create a dashboard component"
System:
  1. Load .ai/agents/react-native-engineer.md
  2. Load required skills
  3. Implement following agent patterns
```

### Rule 2: Always Validate Against Rules

After any agent execution:

```
1. Check all mandatory rules passed
2. Validate architectural boundaries
3. Ensure tests exist (if implementation)
4. Confirm no rule violations
```

### Rule 3: Never Auto-Commit, Auto-Push, or Auto-Merge

Git operations **always** require explicit user confirmation. This includes:

- `git commit` — ask before committing
- `git push` — ask before pushing, even to feature branches
- `gh pr merge` — ask before merging, even when fully approved and CI is green

```
System: "Ready to commit with message: 'feat: add dashboard component'"
System: "Confirm? (yes/no)"
User: "yes"
System: [commits]
```

```
System: "PR #42 is approved and all checks pass. Merge into main? (yes/no)"
User: "yes"
System: [merges]
```

> ⛔ `main` is the only release branch in this project and directly drives App Store / APK builds.
> Merging into `main` without explicit human instruction is a **critical violation**.
> Silence, inferred consent, or "all checks pass" is **never** sufficient authorization.

### Rule 4: Agent Has Authority in Its Domain

Once delegated, the agent makes decisions within its domain:

- `architect` decides architecture
- `reviewer` approves/rejects code
- `quality` determines performance optimizations and Sonar fixes

System orchestrator does not override agent decisions.

### Rule 5: Create Agents for Recurring Patterns

If you handle the same type of request 3+ times without an agent:
→ Create a dedicated agent for that pattern
